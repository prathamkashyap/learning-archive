# Project Explanation Template

## Context

Project explanations should be concrete: problem, users, architecture, trade-offs, and what I personally built.

## One-Minute Version

```text
I built [project] to solve [problem] for [users].
The main features are [features].
The stack is [stack] because [reason].
I worked on [specific contribution].
The hardest part was [challenge], and I solved it by [solution].
```

## Deep-Dive Structure

| Section | Notes |
| --- | --- |
| Problem | What did it solve? |
| Users | Who uses it? |
| Features | What can users do? |
| Architecture | Frontend, backend, database, APIs |
| Data model | Main entities and relationships |
| Trade-offs | What was chosen and why |
| Testing | How correctness was checked |
| Deployment | Where/how it runs |
| Future work | What would improve next |

## Questions to Prepare

- Why did you choose this stack?
- What was the hardest bug?
- How does authentication work?
- How is data stored?
- How would you scale it?
- What would you change if rebuilding it?

## Common Mistakes

- Saying "we" when asked what I personally did.
- Listing features without explaining architecture.
- Overclaiming scale or complexity.
- Not knowing the database schema.

## Quick Reference

| Need | Answer with |
| --- | --- |
| Contribution | Specific files/features owned |
| Challenge | Bug, design trade-off, scaling issue |
| Improvement | Concrete next step |
| Scale answer | Bottleneck and planned fix |
